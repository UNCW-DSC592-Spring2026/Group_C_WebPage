Section 01

Project Overview

The App Review Rating Prediction System is a machine learning solution built to analyze large volumes of free-text app reviews and predict rating outcomes in a structured, interpretable format — replacing a weak baseline model with a well-engineered, fully documented pipeline.

The system moves from a raw 0–1 numeric scale to a simplified 3-class star rating: High (4–5★), Medium (3★), and Low (1–2★). This makes predictions far easier for non-technical stakeholders to interpret and act upon. Longer, more informative reviews are weighted more heavily, while short, sarcastic, or misleading reviews are handled as documented limitations.

Beyond raw predictions, the system surfaces actionable analytics — top-rated apps, most common complaints per app, app rankings, and grouped performance summaries — delivered through a clean web interface accessible to non-technical users.

Section 03

Requirements

Requirements were gathered through stakeholder discussions and categorized into two types. Functional requirements define what the system must do; non-functional requirements define how well it must perform and how it must behave as a system.

⚙️ Functional Requirements
F1
Accept free-text app reviews as input, handling varied length, writing style, and language quality without breaking
F2
Internally preprocess input text (tokenization, stopword removal, normalization) before prediction
F3
Predict a 1–5 star rating and classify each review as High (4–5★), Medium (3★), or Low (1–2★)
F4
Use batch processing to efficiently handle large datasets
F5
Weight longer, more informative reviews more heavily than short or low-signal ones
F6
Rank and group apps by predicted rating for charting and table display
F7
Surface top 10 best/worst apps, most common complaints, and most cited strengths for top-rated apps
F8
Display evaluation metrics (accuracy, F1, confusion matrix) on the interface for transparency
📋 Non-Functional Requirements
N1
Accuracy: Target 90%; minimum acceptable 80% prediction accuracy
N2
Performance: Process 100,000+ reviews within 10–15 minutes using batch processing
N3
Scalability: Support at least 300 concurrent users
N4
Reproducibility: Same dataset must always produce identical predictions (fixed seeds, logged run IDs)
N5
Usability: Interface must be accessible to non-technical users with no ML background
N6
Documentation: All steps tracked via GitHub Issues, PRs, and experiment logs throughout all phases
N7
Transparency: System limitations (sarcasm, short reviews, class imbalance) must be clearly documented

Section 07

Data Exploration

The dataset is an Amazon-style app review collection with 6 columns. The most critical are the free-text review, the numeric rating (0–1 in 0.1 increments), and the AppID for grouping by app. Early exploratory data analysis revealed two major challenges that directly shaped our modeling and preprocessing decisions.

6
Dataset Columns
60.9%
Reviews at Max Rating
3×
Top vs. 2nd Most Common Rating
⚠️
Majority of Reviews Are Very Short
Key Findings & Implications
  • 📊 Severe class imbalance: The maximum rating (1.0) accounts for 60.9% of all reviews — more than 3× the next most common rating. A naive model could achieve high accuracy by predicting "High" for everything. This makes per-class F1, precision, and recall essential for meaningful evaluation.
  • ✂️ Short review prevalence: Most reviews are very brief — some only a few words. This limits the sentiment signal NLP models can extract and directly supports our design decision to down-weight short reviews in the prediction model.
  • 🔢 Rating scale conversion: The raw dataset uses a 0–1 scale. We predict on this original scale for accuracy, then convert to 1–5 stars for the interface — maintaining prediction integrity while meeting stakeholder usability expectations.
  • 🧹 textAnalytics column mostly empty: This column (a 1–100 positivity score) has NaN values for nearly all rows. Cleaning it is required before the pipeline can run, and improving it is a target for Phase 3 experimentation.
  • 🔁 Duplicate review risk: The reviewer name column enables detection and removal of duplicate reviews, preventing a single user from disproportionately influencing a single app's predicted rating.
Section 08

Model & Pipeline Plan

The baseline uses an Azure ML Designer pipeline with a Linear Regression model predicting a continuous rating. Our analysis confirms this is a poor fit — the task is better understood as a 3-class classification problem, and regression metrics obscure the class imbalance issue. Phase 3 replaces and improves this baseline through structured experiments.

Planned Pipeline Components
1
Data Input & Binning

Upload review data and apply a Binning Data component to convert continuous 0–1 ratings into Low / Medium / High categories — the foundation for the classification approach.

3
Vowpal Wabbit Model

Replace Linear Regression with Vowpal Wabbit — an open-source NLP/ML library known for speed and scalability. It handles feature extraction, training, and prediction in one component, making it well-suited for our 100k-review performance target.

4
Data Partitioning & Scoring

Use Partition Data instead of a simple train-test split to validate the model, then score predictions and store them in a database for frontend delivery.

5
Frontend Output

Deliver stored predictions to the website interface as charts, ranked app tables, and complaint summaries. Ratings are scaled to 1–5★ at this stage for stakeholder readability.

Experiment Results — Phase 3

Starting from the baseline Linear Regression pipeline, we made three major architectural changes: binning continuous ratings into three classes (Low ≤ 0.5, Mid = 0.6, High ≥ 0.8), replacing Linear Regression with Vowpal Wabbit for native text classification, and adding Python scripts to handle input formatting and classification output. The built-in Preprocess Text module was removed since Vowpal Wabbit handles text processing internally.

Experiment Progression
1
Initial Vowpal Wabbit Model

First successful run with basic VW settings achieved 83.6% accuracy. However, the Mid class performed exceptionally poorly — the model was heavily biased toward predicting High.

2
Hyperparameter Tuning (~8 variants)

Tested variations in learning rate, number of passes, and n-gram settings. None outperformed the initial basic model — simpler inputs consistently matched or beat more complex configurations.

3
Downsampling the High Class

Attempted to fix class imbalance by undersampling High-rated reviews. Accuracy dropped to a maximum of 77.7%, and Low/Mid classes still performed poorly — undersampling alone was insufficient.

4
Combined Resampling (Final Approach)

Downsampled High class and upsampled Low and Mid classes to balance the training data. The first model with this strategy reached 88.8% accuracy. Several subsequent variants with different VW settings returned the same result — confirming stability. We selected the simplest model at this accuracy as our final pipeline.

5
Output to Blob Storage

Added a Python script to export final predictions to Azure Blob Storage, enabling the Power BI dashboard to pull data directly from the pipeline output.

📋 Full experiment logs with per-model settings and metrics are available in the /experiments/ folder on GitHub.

Section 09

Evaluation Metrics

Due to class imbalance in the dataset, model performance is evaluated using both overall accuracy and class-level metrics. The model achieved an accuracy of 88.8% on over 34.5K reviews, indicating strong performance.

The model performs best on High (4–5 stars) reviews, followed by Low (1–2 stars), while Mid (3 stars) reviews are more difficult to classify. A confusion matrix is used to analyze prediction errors, which mostly occur between similar rating categories. The dashboard also highlights rating distribution and identifies low-rated apps, helping stakeholders understand user feedback and areas for improvement.
